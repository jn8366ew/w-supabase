# GroupMeet MVP PRD

## 개발 전략 (필독)

**UI-First 개발 순서를 엄격히 따를 것:**

1. **Phase 1 — UI/UX Mock 구현**: 모든 페이지와 컴포넌트를 mock data(하드코딩된 더미 데이터)로 먼저 구현한다. Supabase 연동 없이 UI만 완성한다.
2. **Phase 2 — 검토 및 보완**: Mock UI를 실제로 사용해보며 UX 흐름, 레이아웃, 컴포넌트 구조를 검토하고 수정한다.
3. **Phase 3 — DB 설계 확정**: Mock UI 기반으로 확정된 데이터 구조로 Supabase DB 스키마를 설계한다.
4. **Phase 4 — Supabase 연동**: 확정된 스키마로 DB를 생성하고, mock data를 실제 Supabase 데이터로 교체한다.

**태스크 생성 규칙:**

- DB 스키마, TypeScript 타입 생성, Supabase 연동은 반드시 UI Mock 구현 완료 후 진행한다.
- 각 페이지는 mock data 구현 → 검토 → Supabase 연동 순서로 세분화한다.
- 의존성(dependencies)이 이 순서를 반영해야 한다.

## 핵심 정보

**목적**: 카카오톡 기반 소규모 모임 운영자가 참여자 모집, 공지, 정산을 링크 하나로 처리할 수 있도록 지원
**사용자**: 5~20명 규모 소규모 모임을 개인적으로 주최하는 운영자 및 로그인 없이 신청하는 참여자

## 사용자 여정

### 운영자 여정

```
1. 로그인 페이지
   ↓ 카카오 또는 구글 소셜 로그인

2. 대시보드 페이지
   ↓ "모임 만들기" 버튼 클릭 → 모임 생성 폼 제출

3. 모임 상세 페이지 (생성 직후 이동)
   ↓ 초대 링크 복사 → 카카오톡 등으로 공유

   참여자가 링크 클릭 → 참여 신청 페이지 (별도 흐름)

   ↓ 공지 작성 (이메일 발송 여부 선택)
   ↓ 참여자 현황 요약 확인
   ↓ "관리" 버튼 클릭

4. 모임 관리 페이지
   ↓ 참여자 전체 목록 확인
   ↓ 참석 체크 / 정산 처리
   ↓ 완료 후 모임 상세 페이지로 돌아가거나 대시보드로 이동
```

### 참여자 여정

```
1. 카카오톡 초대 링크 클릭
   ↓ 자동으로 참여 신청 페이지로 이동

2. 참여 신청 페이지
   ↓ 모임 정보 확인 (제목, 날짜, 장소, 회비)
   ↓ 이름 입력 (필수)
   ↓ 이메일 입력 + 알림 동의 (선택)
   ↓ "신청하기" 클릭

3. 신청 완료 → 동일 페이지에서 신청 취소 가능
```

| **F007** | 참여자 목록 조회 및 상태 관리 | 참여자 전체 목록, 상태(신청/취소/참석) 필터 및 표시         | 운영자 핵심 작업인 명단 관리               | 모임 관리 페이지 |
| **F010** | 운영자 소셜 인증              | 카카오/구글 OAuth 로그인, Supabase Auth 기반                | 운영자 전용 기능 보호를 위한 최소 인증     | 로그인 페이지    |
| **F011** | 모임 목록 조회                | 운영자가 생성한 모임 목록 카드 형태로 표시                  | 운영자가 여러 모임을 관리하기 위한 기본 뷰 | 대시보드 페이지  |

### 2. MVP 필수 지원 기능

| ID       | 기능명           | 설명                                                      | MVP 필수 이유                       | 관련 페이지      |
| -------- | ---------------- | --------------------------------------------------------- | ----------------------------------- | ---------------- |
| **F005** | 공지 작성        | 텍스트 공지 작성, 참여 신청 페이지에 노출                 | 운영자→참여자 정보 전달의 핵심 수단 | 모임 상세 페이지 |

| **F008** | 참석 체크        | 운영자가 참여자별 실제 참석 여부를 체크박스로 변경        | 행사 당일 현장 운영에 필수          | 모임 관리 페이지 |

## 정산 규칙

- 총 입금액은 `payment_status = paid` 인 참여자의 수 × 회비로 계산한다.
- 취소자(`status = canceled`)는 정산 요약에서 제외한다.
- 참석 여부(`attended`)와 입금 여부(`payment_status`)는 별도로 관리한다.

## 데이터 모델

| 필드       | 설명                            | 타입/관계             |
| ---------- | ------------------------------- | --------------------- |
| id         | 고유 식별자 (Supabase Auth UID) | UUID                  |

| name       | 표시 이름                       | text                  |
| provider   | 소셜 로그인 제공자              | text (kakao / google) |
| created_at | 생성 일시                       | timestamptz           |

### events (모임)

| 필드       | 설명           | 타입/관계   |
| ---------- | -------------- | ----------- |
| id         | 고유 식별자    | UUID        |
| host_id    | 운영자 ID      | → users.id  |
| title      | 모임 제목      | text        |
| date       | 모임 날짜/시간 | timestamptz |
| location   | 장소           | text        |
| fee        | 회비 (원 단위) | integer     |
| capacity   | 모집 인원      | integer     |
| created_at | 생성 일시      | timestamptz |

### participants (참여자)

| 필드            | 설명                       | 타입/관계                                  |
| --------------- | -------------------------- | ------------------------------------------ |
| id              | 고유 식별자                | UUID                                       |
| event_id        | 참여 모임 ID               | → events.id                                |
| name            | 참여자 이름                | text                                       |
| email           | 이메일 (선택 입력)         | text nullable                              |

| payment_status  | 입금 상태                  | enum (unpaid / paid, default unpaid)       |
| created_at      | 신청 일시                  | timestamptz                                |

## MVP 단순화 원칙 (추가)

- `participants.status` 는 `applied / canceled`만 사용한다.
- 실제 참석 여부는 `attended` 필드로 별도 관리한다.
- `capacity`는 MVP에서 표시용 정보이며 신청 제한에는 사용하지 않는다.
- 공지 이메일은 `is_email_opt_in = true` 이고 `status = applied` 인 참여자에게만 발송한다.
- 참여 취소는 MVP에서 신청 직후 동일 세션에서만 지원한다.
- 총 입금액은 `payment_status = paid` 인 참여자 수를 기준으로 계산하며, 취소자는 정산 요약에서 제외한다.
